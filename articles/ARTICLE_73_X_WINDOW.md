# 仕組みを提供する、ポリシーは提供しない——40年動き続けた非同期プロトコル

## はじめに

GitHubにミラーがある。

**mirror/xserver**。X.Org Xサーバーの公式ミラー。`git://anongit.freedesktop.org/git/xorg/xserver` が本家で、実際の開発はfreedesktop.orgのGitLabで続いている。コードベースは巨大だ。しかしその核心——X11プロトコルの設計思想——は、1984年の最初のバージョンから変わっていない。

**X Window System**。1984年6月19日、MIT（マサチューセッツ工科大学）のBob ScheiflerがX Window Systemを正式発表した。それから40年、LinuxのデスクトップはXの上に乗り続けた。Waylandへの移行が本格化しつつある今、Xはようやくメンテナンスモードに入った。

それでも**X11プロトコルは変わっていない**。1987年のX11R1から仕様がロックされ、以来バージョン11のまま今日に至る。40年間、同じプロトコルで動いている。

## W Window Systemの「W」を盗んだ男

1984年。Bob ScheiflerはMIT LCS（Laboratory for Computer Science）の研究者だった。

スタンフォード大学ではすでに**W Window System**が動いていた。Paul AsenteとBrian Reidが設計した、V分散OSの上で動くウィンドウシステムだ。同期型プロトコルを使い、ネットワーク透過性を持つ先進的なシステムだった。

Scheiflerは後に、この有名な発言を残している。

> **「Wからかなりのコードを盗み、同期ではなく非同期のインターフェースで囲み、Xと名付けた。全体的なパフォーマンスはWの約2倍に見える。」**

アルファベットで W の次は X。ネーミングはそれだけだ。

しかし「同期から非同期へ」という転換は、ただのパフォーマンス改善ではなかった。これがXの本質的な設計を決定した。

## 非同期プロトコルの4種類のパケット

X11プロトコルは4種類のパケットだけで構成されている。

```
クライアント（アプリケーション）→ サーバー（ディスプレイ）:

1. Requests（リクエスト）
   - ウィンドウ作成・描画・移動など
   - サーバーはキューに積んで処理
   - 返答を待たない（非同期）

サーバー → クライアント:

2. Replies（返信）
   - QueryWindowなど情報取得リクエストへの応答のみ
   - すべてのリクエストに返信が来るわけではない

3. Events（イベント）  32バイト固定長
   - キーボード入力、マウス移動、ウィンドウ露出など
   - サーバーからクライアントへ非同期に送られる

4. Errors（エラー）  32バイト固定長
   - 不正なリクエストへの応答
   - キューイングされるため、エラー返信は遅延する可能性がある
```

W Window Systemの同期型では、クライアントはリクエストを送るたびに応答を待つ必要があった。Scheiflerの非同期設計では、クライアントはリクエストをキューに積み続け、サーバーはそれを順次処理する。

**ネットワーク遅延が数百ミリ秒あっても、クライアントは止まらない。** これがXをネットワーク越しに使えるシステムにした核心だ。

## クライアントとサーバーの「逆転」

Xには、初心者が必ず一度は混乱するポイントがある。

```
一般的な通念：
  クライアント（ユーザーの端末）← サーバー（リモートで処理）

X Window System：
  X サーバー（ユーザーの端末・ディスプレイ制御）
       ↕
  X クライアント（アプリケーション・ローカル/リモート）
```

**「Xサーバー」はユーザーの端末で動く。** グラフィックスの描画とキーボード・マウスの入力を担当する「表示サービスの提供者」がサーバーだ。アプリケーションはそのサービスを使う「クライアント」として、同じマシン上でも、ネットワーク越しの別のマシン上でも動く。

この設計が「Network Transparent（ネットワーク透過性）」の正体だ。

```bash
# リモートマシン（heavy-server）のアプリケーションを
# ローカルのディスプレイに表示する
ssh heavy-server
export DISPLAY=local-machine:0
xterm &  # ローカルのディスプレイにxtermが開く
```

アプリケーションはどこで動いているかを意識しない。TCPポート6000+ディスプレイ番号で通信する相手がXサーバーであれば、それがローカルでもリモートでも同じコードで動く。

これは1984年の時点で、**クラウドコンピューティングの原型的な概念**だった。

## 「仕組みを提供する、ポリシーは提供しない」

ScheiflerとGettysは1984年の論文で、Xの設計哲学を明文化した。

> **"Provide mechanism, not policy."**
> （仕組みを提供する、ポリシーは提供しない。）

これは宣言であり、警告でもあった。

具体的に何を意味するか。Xはウィンドウを「どう動かすか」の仕組みを提供する。しかしウィンドウが「どう振る舞うべきか」（フォーカスポリシー、ウィンドウ装飾、タイトルバーの位置）は提供しない。それはウィンドウマネージャー（クライアント）が決める。

```
X が提供するもの：
  - ウィンドウの作成・破壊
  - ピクセルの描画
  - キーボード・マウスイベントの配送
  - ウィンドウの移動・リサイズの仕組み

X が提供しないもの：
  - どのウィンドウがフォーカスを持つか
  - タイトルバーをどう描くか
  - ウィンドウをドラッグしたときの動作
  - スクロールバーの外観
```

この設計が、GNOME、KDE、Xfce、twm——まったく異なるデスクトップ環境の共存を可能にした。全員が同じXの上に乗りながら、まったく異なるUIを実現できる。

反面、この設計の批判も有名だ。Don Hopkinsが書いた「The X-Windows Disaster（1994年）」は今も読まれ続けている。「ポリシーを排除した結果、アプリケーション間の一貫性がない」——KDE、GNOME、GTKアプリが混在すると、ウィジェットの見た目が揃わない現象は今日もLinuxデスクトップで起きる。

設計原則は正しかったが、その帰結も引き受けた。

## Project Athena——IBM、DEC、MITの政治学

XはProject Athena（MIT・DEC・IBM共同プロジェクト、1983年〜）の中で生まれた。

目的は「ヘテロジニアスなコンピュータ環境を大学中の学生に均等に提供する」こと。MITのキャンパスにはDECのVAX、IBMのPC、Apolloのワークステーションが混在していた。グラフィカルなUI環境が、どのマシンでも、ネットワーク越しでも動く必要があった。

この「異種システム統合」という要求が、Xのネットワーク透過性設計を必然にした。

1985年、DEC、Hewlett-Packard、Sun Microsystems、ApolloがそれぞれのワークステーションにXをポーティングした。**1986年、MITはXをMITライセンスで公開した。** 特許なし、使用料なし。これが決定的だった。

各社がXを取り込み、Unix GUIの標準になった。IEEEのPOSIX、Open Software Foundation（OSF/Motif）、X/Openも採用。**1992年にはX Consortium（非営利法人）が設立され、MITから権利を引き継いだ。**

## 40年間固定されたプロトコル

X11R1は1987年9月15日にリリースされた。

プロトコルバージョンは「11」。このバージョン番号は以来変わっていない。X11R2、X11R3、...X11R6、X11R6.9——「R（Release）」の数字は増えても、プロトコル自体は変わっていない。

```c
/* xproto.h - X11プロトコル定義
 * 1987年以来、このヘッダーの基本構造は変わっていない
 */
#define X_PROTOCOL      11      /* current protocol version */
#define X_PROTOCOL_REVISION 0   /* current minor version */

typedef struct {
    CARD8   reqType;
    CARD8   data;       /* data field, or 0 */
    CARD16  length;     /* length in 4-byte units of request */
} xReq;
```

40年間、同じリクエスト構造で通信している。

**X11R6.9（2005年12月21日リリース）が事実上の最終版だ。** X11R7.0は同じソースコードをビルドシステムだけ変えたもの（imake→自動ツール）。それ以降、新機能追加は基本的に行われていない。

## Waylandへの移行と40年の清算

2024年、X.Orgはメンテナンスモードへの移行を宣言した。

**Wayland**は2008年に開発が始まった代替プロトコルだ。Xのネットワーク透過性は捨て、その代わりに：
- コンポジタ（画面合成）をプロトコルに組み込む
- セキュリティモデルを近代化（アプリが他のアプリのウィンドウにアクセスできない）
- 遅延を削減（XWaylandのような中間レイヤーが不要）

主要なLinuxディストリビューションはほぼすべてWaylandをデフォルトにした。KDE Plasma 6.8ではX11セッションの廃止が予定されている。

しかし**XWayland**が存在する。WaylandコンポジタがXクライアントとの互換性を維持するための翻訳レイヤーだ。40年間のXアプリケーションが、XWaylandを通じてWaylandの上で動く。

これは遺言状のようなものだ。Xは死ぬが、Xのクライアントは生き続ける。XWaylandがそれを保証している。

## 鑑定

```
リポジトリ : gitlab.freedesktop.org/xorg/xserver（mirror/xserver）
言語       : C
発表日     : 1984年6月19日（Bob Scheifler、MIT）
核心技術   : 非同期X11プロトコル（1987年固定）
設計哲学   : "Provide mechanism, not policy"
誕生の場   : MIT Project Athena（DEC・IBM共同出資）
ライセンス : MITライセンス（1986年〜）
現状       : メンテナンスモード（2024年〜）、XWaylandで継続
```

ScheiflerはWから「かなりのコードを盗んだ」と言った。盗んだだけではなく、非同期に変えた。それが40年持つ設計になった。

「仕組みを提供する、ポリシーは提供しない」——この原則がGNOMEとKDEの並立を可能にし、同時にLinuxデスクトップの「バラバラ感」の源でもあった。設計とは常に、何かを得るために何かを捨てることだ。

Waylandはポリシーをプロトコルに組み込んだ。Xが捨てたものを、Waylandは取り戻した。40年後の清算だ。

しかし**X11プロトコルは今も動いている**。XWaylandの中で。1987年の仕様のまま。
