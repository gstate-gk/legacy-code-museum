# Unixを作った男たちが「Unixの次」を作った——そして世界はUnixを使い続けた

## はじめに

GitHubの片隅に「理想」のコードを見つけた。

**Plan 9 from Bell Labs**。1992年、Rob Pike と Ken Thompson が C言語で書いたオペレーティングシステムだ。

数十万行。C 71%、アセンブリ 4%。2,483コミット。**Unix を作った人間たちが、Unix の「次」として設計した OS。**

名前は Ed Wood の1957年のカルト映画 *Plan 9 from Outer Space* から取られた。「宇宙からの計画9号」——史上最低の映画と称される作品から、史上最も野心的な OS が名を借りた。

Plan 9 は Unix の反省から生まれた。Unix は「全てはファイルである」と言いながら、ネットワーク、グラフィックス、プロセス間通信は特別扱いだった。Plan 9 は **本当に全てをファイルにした。** ネットワークインターフェースは `/net/ether0`。マウスは `/dev/mouse`。プロセスの状態は `/proc` のファイル。ウィンドウシステムすら `/dev/draw` へのファイル操作。

そして Plan 9 は、世界で最も重要な文字エンコーディングを生んだ。**UTF-8**——ニュージャージーのダイナーで、紙ナプキンの上に描かれた設計図から。

## 発掘された痕跡

### 痕跡1：本当に全てがファイルになった

```c
enum{
    Qdir,
    Qtrace,
    Qargs,      /* process arguments */
    Qctl,       /* control operations */
    Qfd,        /* file descriptors */
    Qmem,       /* memory */
    Qregs,      /* registers */
    Qfpregs,    /* floating point registers */
    Qstatus,    /* process status */
    Qsegment,   /* memory segments */
    Qnote,      /* notifications */
    Qwait,      /* wait for child */
    Qns,        /* namespace */
    Qtext,      /* executable text */
};
```

`devproc.c`——`/proc` ファイルシステムの実装。プロセスの全ての情報がファイルとして公開される。引数、制御、メモリ、レジスタ、ステータス、名前空間——**全てが `read` と `write` で操作できる。**

Unix の `/proc` は後付けだった。Plan 9 の `/proc` は設計の根幹だ。プロセスをデバッグしたければ、`/proc/PID/mem` を読めばいい。プロセスを殺したければ、`/proc/PID/ctl` に `kill` と書き込めばいい。`kill` コマンドは不要。ファイル操作だけで全てが完結する。

VPN が欲しければ？リモートサーバーの `/net/ether0` を自分の `/net/ether1` にマウントする。**VPN ソフトウェアは不要。ファイルシステムの操作だけで VPN が実現する。**

### 痕跡2：ダイナーの紙ナプキンから生まれた UTF-8

1992年9月のある水曜日の夜。Ken Thompson と Rob Pike はニュージャージーのダイナーにいた。

X/Open グループからの電話があった。テキサスのオースティンで会議中の彼らが、Unicode エンコーディングの提案について意見を求めてきた。Thompson と Pike は既存の提案を嫌っていた——ヌル文字が含まれるため、C の文字列関数が壊れる。

**Thompson はダイナーのプレースマット（紙ナプキン）の上に、新しいエンコーディングの設計を描いた。**

- 水曜日の夜：設計完了
- 木曜日：コーディング完了
- 金曜日：Plan 9 全体を新エンコーディングに移行完了
- 月曜日：X/Open グループが投票で採用

**UTF-8 は2日で実装され、3日目に OS 全体が移行した。** Pike は「あのプレースマットを取っておけばよかった」と後悔している。

2026年現在、**Web 上のテキストの98%以上が UTF-8** でエンコードされている。ダイナーの紙ナプキンの上に描かれた設計が、インターネットの文字エンコーディングの標準になった。

### 痕跡3：`rc` シェル——C シェルへの反論

```c
char *argv0="rc";

void
start(code *c, int pc, var *local)
{
    struct thread *p = new(struct thread);
    p->code = codecopy(c);
    p->pc = pc;
    p->argv = 0;
    ...
    p->ret = runq;
    runq = p;
}
```

`rc/exec.c`——Plan 9 のシェル `rc`（run commands）。Unix の Bourne シェルや C シェルの複雑さを否定し、**最小限の構文で最大限の機能を提供する。**

`rc` は後に Unix にも移植され、Plan 9 が死んだ後も生き続けている。

### 痕跡4：`cat -v` は有害である

Plan 9 の設計思想を象徴する論文がある。Rob Pike による **「cat -v Considered Harmful」**（1983年）。

Unix の `cat` コマンドは、本来ファイルを連結するだけのツールだったはずが、BSD Unix の下で `-v`（非表示文字の可視化）、`-n`（行番号）、`-s`（空行の圧縮）などのオプションが追加された。Pike はこれを「ソフトウェアの腐敗」と呼んだ。

Plan 9 の `cat` は純粋だ。ファイルを連結する。それだけ。行番号が欲しければ別のツールを使え。**一つのプログラムは一つのことだけをする。** Unix が忘れかけていた哲学を、Plan 9 は取り戻そうとした。

### 痕跡5：9P プロトコル——全てを繋ぐ糸

Plan 9 の全てのサービスは **9P プロトコル** で通信する。ファイルサーバー、ウィンドウシステム、ネットワーク、デバイス——全てが同じプロトコルで話す。

ウィンドウマネージャ `rio` は、子プロセスに `/dev/draw` のような仮想ファイルをマウントする。GUI プログラムは、自分がフレームバッファ上で直接動いているのか、ウィンドウ内で動いているのかを **知らない。知る必要がない。** ファイルに書き込めば、適切な場所に描画される。

2026年現在、9P プロトコルは Linux カーネルに組み込まれ、WSL（Windows Subsystem for Linux）でも使われている。**Plan 9 は死んだが、9P は生きている。**

### 痕跡6：Go 言語——Plan 9 の魂の転生

Plan 9 は商業的に失敗した。しかしその魂は **Go 言語** に転生した。

Go は2009年に Google で生まれた。設計者は Rob Pike、Ken Thompson、Robert Griesemer。Pike と Thompson は Plan 9 の中心人物だ。

Go のツールチェーンは Plan 9 のツールチェーンの直系の子孫。Go の C方言は Plan 9 の C方言の進化形。ゴルーチンとチャネルによる並行処理は、Plan 9 が追求した分散コンピューティングの思想の延長線上にある。

**Plan 9 は OS として失敗したが、プログラミング言語として復活した。**

## なぜ世界は Unix を使い続けたのか

Eric S. Raymond は書いた——**「Unix は軋み、きしむ。しかし十分に仕事をこなし、そのポジションを守っている」。**

Plan 9 は正しかった。全てをファイルにする思想は美しい。9P プロトコルは優雅だ。名前空間はエレガントだ。

しかし「正しい」ことと「使われる」ことは違う。Plan 9 にはデバイスドライバが足りなかった。アプリケーションが足りなかった。商用サポートが足りなかった。そして Unix は「十分に良かった」。

Richard Gabriel の「Worse is Better」論文（1989年）が全てを予言していた。**「正しさ」より「実装の単純さ」が勝つ。** Multics（鑑定書 #034）は Plan 9 と同じ運命を辿った——正しすぎて、複雑すぎて、Unix に負けた。Plan 9 は **Unix から生まれ、Unix に負けた。** 皮肉な歴史の反復だ。

## 推定される経緯

**1980年代後半**: Bell Labs の Computing Science Research Center で開発開始。

**1992年**: 大学向けに初回リリース。同年9月、Thompson と Pike がダイナーで UTF-8 を設計。

**1995年**: Harcourt Brace を通じて商用リリース。

**2000年**: オープンソースライセンスで公開。

**2002年**: 第4版（最終安定版）リリース。

**2011年**: Dennis Ritchie 死去。Bell Labs の OS 研究の時代が終わる。

**2009年**: Go 言語発表。Pike と Thompson が Plan 9 の思想を言語として再実装。

**2021年3月23日**: 著作権が Bell Labs から Plan 9 Foundation に移転。開発再開。

**2026年現在**: 9front コミュニティが開発を継続。9P プロトコルは Linux、WSL で使用中。

### 希少度: ★★★★☆
C言語だが Plan 9 方言（独自拡張）。名前空間、9P プロトコル、`/proc` ファイルシステム——設計思想そのものが希少。OS としてはほぼ絶滅しているが、コードは Plan 9 Foundation を通じて保存されている。

### 技術的負債密度: ★☆☆☆☆
**ほぼゼロ。** Plan 9 は「正しく」設計された OS だ。Unix が積み上げた技術的負債を清算するために作られた。`cat -v Considered Harmful` の精神が全コードに行き渡っている。負債が少なすぎたことが、皮肉にも普及を妨げた——「十分に汚い」Unix の方が、実世界では使いやすかった。

### 考古学的価値: ★★★★★

Unix の作者が Unix を否定する物語。ダイナーの紙ナプキンで UTF-8 が生まれる夜。「正しいもの」が「十分に良いもの」に負ける皮肉。Go 言語による魂の転生。そして Eric Raymond の予言——「Unix は軋むが、仕事はする」。

## 鑑定人所見

Plan 9 は「設計図」だ。

Unix を作った男たちが描いた、**「Unix はこうあるべきだった」** という設計図。全てをファイルに。全てを 9P で。全てを名前空間で。美しく、正しく、一貫性がある。

しかし世界は設計図の美しさでは動かない。Multics（鑑定書 #034）が「正しすぎて」Unix に負けたように、Plan 9 は「正しすぎて」Unix に負けた。**歴史は繰り返す——理想主義は実用主義に敗れる。**

最も象徴的なのは **UTF-8** だ。ダイナーの紙ナプキンに描かれた設計図から、2日で実装され、3日目に OS 全体が移行した。そして今、Web の98%以上がその設計図の上で文字を表示している。Plan 9 は死んだが、UTF-8 は地球上の全てのブラウザで生きている。

PDP-7 Unix（鑑定書 #043）は「種子」だった——3,000行から全ての OS が生まれた。Plan 9 は **「設計図」** だ——Unix の作者たちが描いた理想の OS の設計図。建物は建たなかったが、設計図の一部は世界中の建物の中に組み込まれている。
