# 駆逐艦のデータベースが落ちるたびにミサイルシステムが止まるので、自分で作った話

## はじめに

GitHubの片隅に「最も使われたソフトウェア」のコードを見つけた。

**SQLite**。2000年、D. Richard Hippが書いた組み込み型リレーショナルデータベースエンジンだ。C言語、約155,000行。サーバー不要。設定不要。インストール不要。単一ファイルにデータベース全体が収まる。

あなたのスマートフォンに入っている。あなたのブラウザに入っている。あなたのPCのOSに入っている。推定**1兆個以上**のデバイスに搭載されている——人類が作ったソフトウェアの中で、最も多くデプロイされたものの一つ。

そしてソースコードの冒頭に、こう書いてある。

```
May you do good and not evil.
May you find forgiveness for yourself and forgive others.
May you share freely, never taking more than you give.
```

「善をなし、悪をなさぬように。自分を許し、他者を許すように。惜しみなく分かち合い、受け取る以上に奪わぬように。」

著作権表示ではない。**祝福**だ。

---

## 駆逐艦から始まった

SQLiteが生まれた理由は、ミサイル駆逐艦だった。

2000年春、D. Richard HippはGeneral Dynamics（防衛企業）の契約で、米海軍ミサイル駆逐艦USS Oscar Austin（DDG-79）のダメージコントロールシステムを開発していた。艦内のパイプ、バルブ、ポンプを管理するシステムだ。

問題は**Informix**だった。

システムのバックエンドにInformixデータベースサーバーが使われていた。Informixが落ちるたびに、駆逐艦のダメージコントロールシステムが止まる。**戦闘中にデータベースが落ちたら、パイプの損傷箇所がわからなくなる。**

Hippはこう考えた——「サーバーが必要ないデータベースを作ればいい」。

2000年8月17日、SQLite 1.0をリリース。900KB未満。サーバーレス。ノーインストール。パブリックドメイン。

駆逐艦のためのデータベースが、24年後に1兆個のデバイスに搭載されることになる。

---

## 発掘された痕跡

### 痕跡1：「著作権ではなく祝福」——パブリックドメインへの献呈

```
May you do good and not evil.
May you find forgiveness for yourself and forgive others.
May you share freely, never taking more than you give.

The author disclaims copyright to this source code.
```

GPLでもMITでもBSDでもない。**パブリックドメイン**——著作権そのものの放棄。

Hippと開発チーム全員が、署名済みの宣誓書を提出している。宣誓書は耐火金庫に保管されている。「著作権を放棄する」という法的行為を、「祝福」という精神的行為として実践した。

なぜパブリックドメインなのか？ Hippは軍の契約で開発を始めた。米国連邦政府の著作物はパブリックドメインだ。その精神を引き継いだ。そしてパブリックドメインだからこそ、Apple、Google、Microsoft、Mozilla——誰でも自由にSQLiteを組み込める。ライセンス交渉もロイヤリティもない。1兆個のデプロイを可能にしたのは、この「祝福」だ。

### 痕跡2：テスト590倍——人類が書いた最もテストされたコード

SQLiteのテスト体制は異常だ。

| 項目 | 行数 |
|:---|---:|
| コアライブラリ | 155,800行 |
| テストコード | **92,053,100行** |
| **比率** | **590倍** |

本体の**590倍**のテストコード。テストがソースの590倍ある。

4つの独立したテストハーネス：
- **TCLテスト**: 51,445テストケース
- **TH3テスト**: 50,362テストケース、リリース前に2億4,850万テスト実施
- **SQLLogicTest**: 720万SQLクエリ
- **dbsqlfuzz + jfuzz**: 毎日約5億テストケース実行

100%分岐カバレッジ（MC/DC）を達成。航空機のフライトソフトウェア基準**DO-178B**に準拠。実際に航空機のフライトソフトウェアにSQLiteは搭載されている。

異常系テスト：
- メモリ枯渇（OOM）シミュレーション
- ディスク満杯シミュレーション
- 電源喪失シミュレーション
- コズミックレイレベルのビット反転検証

Hippが100%カバレッジを達成するために集中したのは1年間。その結果、**以降8〜9年間、ほぼバグ報告がなくなった**。

### 痕跡3：「fopen()の代替」——設計哲学の宣言

SQLiteの本質を最も正確に表す一文がある。

> *「Think of SQLite not as a replacement for Oracle but as a replacement for fopen().」*
> 「SQLiteはOracleの代替ではない。fopen()の代替だ。」

fopen()——C言語のファイルオープン関数。SQLiteは「データベースサーバー」ではなく、「ファイルI/Oの高レベルな代替手段」なのだ。

- サーバーがない（fopen()にサーバーはいらない）
- インストールがない（fopen()にインストールはいらない）
- 設定がない（fopen()に設定はいらない）
- 単一ファイル（fopen()も単一ファイル）

そしてもう一つの宣言。

> *「Small. Fast. Reliable. Choose any three.」*
> 「小さい。速い。信頼性が高い。3つ全部選べ。」

通常の工学的トレードオフでは「3つのうち2つを選べ」だ。SQLiteは3つ全部を達成したと宣言している。590倍のテストが、その自信の根拠だ。

### 痕跡4：1兆個のデバイス——最も多くデプロイされたソフトウェア

SQLiteが搭載されている場所：

- **全てのAndroidスマートフォン** — 連絡先、SMS、設定、アプリデータ
- **全てのiPhone** — 同上
- **全てのWindows 10/11** — 内蔵
- **全てのmacOS** — 内蔵
- **Chrome、Firefox、Safari** — ブラウザ内蔵
- **Skype、iTunes、Dropbox** — アプリ内蔵
- **Python、PHP** — 言語ランタイム内蔵
- **航空機のフライトソフトウェア** — DO-178B準拠

計算：スマートフォン40億台 × 各端末に数百のSQLiteデータベース = **1兆個以上**。

zlibに次ぐ「2番目に広く配布されたソフトウェアコンポーネント」と推定されている。

### 痕跡5：Amalgamation——15万行を1つのCファイルに

SQLiteには「amalgamation（統合）」という配布方式がある。

通常、155,800行のソースコードは複数のファイルに分割されている。しかしSQLiteは**全コードを1つの巨大なCファイル（sqlite3.c）に統合**して配布する。

なぜか？

- コンパイラの最適化が効きやすい（ファイル間のインライン展開）
- 約5%のパフォーマンス向上
- ビルドが1コマンドで完了（`gcc sqlite3.c -o sqlite3`）
- 組み込みデバイスへの搭載が極めて簡単

15万行の1ファイル。普通のソフトウェア工学では「アンチパターン」だ。しかしSQLiteの用途——組み込み、配布の容易さ——には完璧に合理的。**常識に逆らう設計が正解になる**稀有な例。

### 痕跡6：VDBE——SQLをバイトコードにコンパイルする

SQLiteの内部アーキテクチャの核心は**VDBE（Virtual Database Engine）**だ。

```
SQL文
  ↓ Tokenizer（字句解析）
  ↓ Parser（構文解析）
  ↓ Code Generator（コード生成）
  ↓ Bytecode（バイトコード）
  ↓ VDBE Interpreter（仮想マシン実行）
結果
```

SQLiteはSQLを直接実行しない。まずバイトコードにコンパイルし、仮想マシンで実行する。Mako VMの32オペコードと同じ発想——ただし、こちらはデータベース操作に特化している。

`EXPLAIN`コマンドで生成されたバイトコードが見える。

```sql
EXPLAIN SELECT * FROM users WHERE age > 30;
-- addr  opcode       p1  p2  p3
-- 0     OpenRead     0   2   0
-- 1     Rewind       0   7   0
-- 2     Column       0   1   0
-- 3     Integer      30  0   0
-- 4     Le           0   6   0
-- 5     ResultRow    0   2   0
-- 6     Next         0   2   0
-- 7     Close        0   0   0
-- 8     Halt         0   0   0
```

### 痕跡7：「2050年まで」——25年間の保守コミットメント

公式サイトにこう明記されている。

> *「The future is always hard to predict, but the intent of the developers is to support SQLite through the year 2050.」*
> 「未来を予測することは常に難しいが、開発者の意図はSQLiteを2050年まで支えることだ。」

2050年。SQLiteが生まれてから50年。Hippが100歳を超える年。

> *「Design decisions are made with that objective in mind.」*
> 「全ての設計判断は、その目標を念頭に置いてなされている。」

1兆個のデバイスに搭載されたソフトウェアの開発者が、「2050年まで動き続ける」ことを約束している。パブリックドメインだから、たとえHippがいなくなっても、誰でもメンテナンスできる。祝福は永遠に残る。

## 推定される経緯

**2000年春**: D. Richard HippがGeneral Dynamicsの契約で駆逐艦USS Oscar Austinのソフトウェアを開発。Informixの不安定さに直面。

**2000年8月17日**: SQLite 1.0リリース。gdbmの代替として、SQLインターフェースを持つ組み込みデータベースエンジン。

**2001年**: Motorolaが携帯電話にSQLiteを採用。組み込み市場への進出。

**2004年**: Firefox 1.0がSQLiteを内蔵。ブックマーク、履歴、Cookieの保存に使用。

**2005年**: Apple がMac OS X 10.4でSQLiteを採用。Core Dataの基盤として。

**2007年**: iPhone発売。iOSにSQLiteが搭載される。Androidも同様。全スマートフォンにSQLiteが搭載される時代が始まる。

**2010年**: WAL（Write-Ahead Logging）モード実装。読み書き並行実行が可能に。

**2017年**: 推定デプロイ数が1兆個を突破。

**2026年現在**: 24年間のメンテナンス継続。2050年までの保守を宣言。

## 鑑定人所見

SQLiteは「祝福のコード」だ。

著作権表示の代わりに祝福が書かれている。ライセンス交渉の代わりに宣誓書が耐火金庫に保管されている。利益の代わりに1兆個のデバイスへの無償提供がある。

Richard Hippは駆逐艦のためにデータベースを作った。Informixが落ちるたびにミサイルシステムが止まる問題を解決するために。900KBの解決策は、24年後に人類のインフラになった。

最も印象的なのは**テスト590倍**という数字だ。155,800行の本体に対して、92,053,100行のテスト。これは品質への執着ではなく、**責任**だ。1兆個のデバイスが依存するコードに、バグを入れるわけにはいかない。航空機のフライトソフトウェアに搭載されるコードに、例外は許されない。

そして「2050年まで」。このコミットメントは、パブリックドメインだからこそ成立する。Hippがいなくなっても、SQLiteは死なない。著作権が切れることもない——最初から存在しないのだから。

CP/Mは「技術で勝ってビジネスで負けた」。Xerox PARCは「発明して売れなかった」。SQLiteは第三の道を選んだ——**売らないことで、全てのデバイスに入り込んだ**。

祝福は永遠に残る。1兆個のデバイスの中で。
