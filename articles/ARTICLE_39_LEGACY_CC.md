# 全てのソフトウェアの起源——構造体がまだなかった頃のC言語

## はじめに

GitHubの片隅に「起源」のコードを見つけた。

**legacy-cc**。1972年、ベル研究所の Dennis Ritchie が PDP-11 向けに書いた、現存する最古のCコンパイラのソースコードだ。磁気テープから復元された。

5,163行。2パス構成。自己コンパイル可能。

しかしこのCには、**構造体がない**。`long` もない。`#define` もない。`+=` は `=+` と書く。現代のプログラマーが「C言語」と聞いて思い浮かべるものの多くが、まだ存在しない。

これは C言語の「誕生の瞬間」だ。

そしてこのコードを書いた男は、2011年10月12日に一人で亡くなった。Steve Jobs の死の1週間後。世界中がJobsを追悼する中、**iPhone も Mac も Android も、全てこの男のコードの子孫であることを、ほとんど誰も語らなかった。**

---

## ベル研究所、1972年

Dennis Ritchie はハーバード大学で物理学と数学を学び、1967年にベル研究所に入所した。Ken Thompson と出会い、Multics プロジェクトに参加。AT&T が Multics から撤退した後、Thompson は廃棄された PDP-7 で UNIX を書き始めた。Ritchie はその隣にいた。

Thompson は B 言語を作った。BCPL の影響を受けた、型のない言語だ。しかし B には型がない——int も char もない。全てが「ワード」だ。PDP-11 のバイトアドレッシングを活かすには、型が必要だった。

Ritchie は B に型を加え始めた。最初は "NB（New B）" と呼んだ。やがてそれは "C" になった。

Ritchie 自身の回想——

> *「1972-73年がC言語の形成期だった。型のないBから弱い型付けのCへの移行が行われ、その間にNBという（ネアンデルタール的な？）言語を経由した。NBのソースは残っていないようだ。」*

## 発掘された痕跡

### 痕跡1：コメントアウトされた `long`——C言語の成長が1行で見える

このコンパイラには2つのバージョンが存在する。

**last1120c**（1972年7月頃）——初期バージョン:

```c
init("int", 0);
init("char", 1);
init("float", 2);
init("double", 3);
/*	init("long", 4);  */
```

**prestruct-c**（1972年12月頃）——構造体追加直前:

```c
init("int", 0);
init("char", 1);
init("float", 2);
init("double", 3);
init("struct", 4);
```

`long` はコメントアウトされたまま実装されなかった。数ヶ月後、そのスロット4に `struct` が入った。

Ritchie の注釈——

> *「興味深いのは、初期のコンパイラでは 'long' キーワードの準備がコメントアウトされていること。後のバージョンではそのスロットを 'struct' が引き継いだ。long の実装は数年先のことだった。」*

**C言語の「成長」が、たった1行のコードで可視化される。**

### 痕跡2：`=+` 演算子——現代の `+=` の前身

1972年のCでは、複合代入演算子が逆順だった。

```c
i =+ *sp++;    /* 現代の i += *sp++;  */
i =% hshsiz;  /* 現代の i %= hshsiz; */
i =* pssiz;   /* 現代の i *= pssiz;  */
c =<< 1;      /* 現代の c <<= 1;     */
```

なぜ逆順か？ B言語からの直接的な継承だ。しかしこの記法は致命的な曖昧性を生んだ——

`x=-1` は「x に -1 を代入する」のか、「x から 1 を引く」のか？

この問題が後に `=+` から `+=` への変更を引き起こした。現代のCプログラマーが当然のように使う `+=` は、1972年のバグの記憶から生まれた。

### 痕跡3：`waste()` ——意図的に空間を浪費する関数

コード生成器（c10.c）の冒頭に、驚愕のコードがある。

```c
waste()		/* waste space */
{
	waste(waste(waste),waste(waste),waste(waste));
	waste(waste(waste),waste(waste),waste(waste));
	waste(waste(waste),waste(waste),waste(waste));
	waste(waste(waste),waste(waste),waste(waste));
	waste(waste(waste),waste(waste),waste(waste));
	waste(waste(waste),waste(waste),waste(waste));
	waste(waste(waste),waste(waste),waste(waste));
	waste(waste(waste),waste(waste),waste(waste));
}
```

「空間を浪費する」と明記された関数。自分自身を再帰呼び出しするふりをして、コード領域を意図的に消費する。

なぜか？ PDP-11/20 にはメモリマッピングがない。コンパイラはパス1の初期化コードを実行した後、**その領域を一時記憶域として上書きする**。`waste()` は、上書き可能な領域を十分に確保するための「詰め物」だ。

Ritchie 自身がこれを **"astonishing peculiarity（驚くべき奇妙さ）"** と呼んだ。

### 痕跡4：`ospace() {}` ——「フェイク」と書かれた空の関数

```c
ospace() {}	/* fake */
```

「フェイク」とコメントされた、中身が空の関数。しかしこの関数は重要な役割を果たす——**この関数のアドレスが、一時記憶域の開始点として使われる。**

```c
sp = ospace;  /* 関数名＝そのメモリアドレス */
```

関数名をポインタとして使い、その先のメモリを一時バッファとして再利用する。`waste()` が確保した「ゴミ領域」の先頭が `ospace` のアドレスだ。

現代のプログラマーが書いたら、即座にコードレビューで却下される。しかし1972年、4,096ワードのメモリで動くコンパイラを書くには、こうするしかなかった。

### 痕跡5：5,163行の自己コンパイル——鶏と卵の問題

このコンパイラは **自分自身をコンパイルできる**。しかし復元には「鶏と卵」の問題があった。

1997年、Ritchie は "s1"、"s2" とラベルされた DEC テープを、オーストラリアの Bond 大学の Warren Toomey に寄贈した。s2 テープの `/usr/lib` に、初期Cコンパイラのバイナリ（c0 と c1）が眠っていた。

しかしソースコードは原始的すぎて、現代のCコンパイラではパースできない。`=+` 演算子、宣言の違い、プリプロセッサの不在——全てが壁になった。

Toomey の解決策——PDP-11 のユーザーモード命令をシミュレートする **Apout エミュレータ** を書き、s2 テープのバイナリ上で last1120c のソースをコンパイルした。

**コンパイルされた last1120c は、自分自身を再コンパイルできた。**

Ritchie の「挑発」がきっかけだった——

> *「last1120c コンパイラの動作するバージョンを構築するには、おそらく十分な資料がある。」*

Toomey はこれを「暗黙の挑戦」と受け取った。

### 痕跡6：自前の printf——標準ライブラリがまだ存在しない

1972年、C の標準ライブラリはまだ存在しない。コンパイラは printf を自前で実装している。

```c
case 'p':
    s = x;
    putchar('_');
    c = namsiz;
    while(c--)
        if(*s)
            putchar(*s++);
    goto loop;
```

`%p` という独自フォーマットが、外部シンボル名の出力に使われる。先頭に `_`（アンダースコア）を付加する——これは現代のCでも生き残っている慣習だ。関数名の先頭に `_` が付くのは、1972年のこのコードに起源がある。

### 痕跡7：`++` と `--` の秘密——PDP-11 のために生まれた演算子

C言語の `++` と `--` 演算子は、PDP-11 のアドレッシングモードに直接対応している。

PDP-11 には「オートインクリメント」と「オートデクリメント」のアドレッシングモードがあった。`*(p++)` が文字通り **1つの機械語命令** にコンパイルされる。

```c
/* PDP-11 では *(--sp) = *(p++) が2命令で実行可能 */
```

C の `++` と `--` は「便利な略記法」ではない。**ハードウェアの命令セットそのもの** だ。PDP-11 が別のアーキテクチャだったら、C言語にインクリメント/デクリメント演算子は存在しなかったかもしれない。

## もう一人の創造者

C言語の物語は、Ritchie 一人では語れない。

**Ken Thompson** は UNIX を作り、B 言語を作った。Ritchie は B を C に進化させ、1973年に UNIX を C で書き直した。二人の共同作業は数十年に及んだ。

1983年、二人はチューリング賞を受賞した。Thompson の受賞講演「Reflections on Trusting Trust」で、**Cコンパイラにバックドアを仕込む** という思考実験を披露した——コンパイラが自分自身をコンパイルする時にバックドアを埋め込み、ソースコードを読んでも痕跡が見えない。

**自己コンパイルするコンパイラの信頼性の問題。** これは legacy-cc の「鶏と卵」の問題の、哲学的な裏面だ。

**世界中の全てのスマートフォン、全てのサーバー、全ての組み込みシステムは、1972年にベル研究所で生まれた5,163行のコードの子孫だ。**

## 推定される経緯

**1967年**: Ritchie がベル研究所に入所。Thompson と出会う。

**1969年**: AT&T が Multics から撤退。Thompson が PDP-7 で UNIX を開発開始。

**1969-70年**: Thompson が B 言語を設計。BCPL の影響。型なし。

**1971年**: Ritchie が B に型を追加し始める（NB = New B）。

**1972年7月頃**: **last1120c** 完成。PDP-11/20 向け。構造体なし、long コメントアウト。

**1972年12月頃**: **prestruct-c** 完成。struct キーワード追加、-> 演算子追加。

**1973年**: UNIX Version 4 を C で書き直し。OS がアセンブリからC言語に移行した歴史的瞬間。

**1978年**: Kernighan と Ritchie が『The C Programming Language』出版（K&R）。

**1983年**: Thompson と共にチューリング賞受賞。

**1997年**: Ritchie が DEC テープを Warren Toomey に寄贈。

**2009年**: Toomey が Apout エミュレータで last1120c の自己コンパイルに成功。

**2011年10月12日**: Ritchie 死去。

## 鑑定人所見

legacy-cc は「根」だ。

5,163行。2パス。自己コンパイル可能。構造体がない。`long` がない。プリプロセッサがない。`+=` は `=+` と書く。現代のCプログラマーが見たら、同じ言語とは思えないだろう。

しかしこの5,163行から、全てが始まった。

最も象徴的なのは、**コメントアウトされた `long` が `struct` に置き換わる瞬間**だ。1972年7月の last1120c では `/*init("long", 4);*/`。5ヶ月後の prestruct-c では `init("struct", 4);`。C言語がB言語の延長から独自の言語に成長する瞬間が、たった1行のコードに凝縮されている。

`waste()` 関数は、1972年のプログラミングの現実を突きつける。4,096ワードのメモリでコンパイラを動かすために、初期化コードの領域を一時バッファとして上書きする。現代のコードレビューなら即却下だ。しかし Ritchie はこれを "astonishing peculiarity" と呼んだ——恥じるのではなく、驚くべき工夫として記録した。この知的誠実さは SHRDLU の Winograd（鑑定書 #026）に通じる。

Ritchie 自身がこのコードについて書いた言葉が、全てを物語っている——

> *「この資料を見返すと、複雑な感情がある。多くが未熟で出来が良くなく、公開するのは少し恥ずかしい。しかし同時に、これは創造の時期の2つの瞬間を捉えており、いくらかの歴史的興味があるかもしれない。」*

「いくらかの歴史的興味」。世界中の全てのソフトウェアの起源を、この男は「いくらかの興味」と呼んだ。

2011年10月、Steve Jobs の死の1週間後に、Dennis Ritchie は一人で亡くなった。Rob Pike は言った——「Dennis はもっと大きな影響を与えた。そして世間は彼が誰かすら知らない」。

しかしコードは知っている。全てのスマートフォンの中で、全てのサーバーの中で、全ての組み込みシステムの中で、1972年の5,163行の子孫が今も動いている。

Spacewar!（鑑定書 #027）は「遊び場」だった。legacy-cc は **「根」** だ——目に見えないが、全ての木を支えている。
