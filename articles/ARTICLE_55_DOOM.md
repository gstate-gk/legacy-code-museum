# 「MONDO HACK!」——NeXTの上で地獄を作り、ピザ代でフェラーリを買った男

## はじめに

GitHubの片隅に「地獄」のコードを見つけた。

**DOOM**。1993年、John Carmack が C言語で書いたファーストパーソンシューティングゲームだ。

約39,000行（linuxdoom-1.10）。C++ 96%、C 4%。**エンジン全体を Carmack が一人で書いた。**

前作 Wolfenstein 3D（鑑定書 #033）は「扉」だった——3D世界への扉。DOOM はその扉をくぐった先にある **地獄** だ。Wolfenstein 3D が90度の壁と均一な照明しか描けなかったのに対し、DOOM は斜めの壁、高低差のある床と天井、可変照明を実現した。しかも、386 の 16MHz CPU で。

開発は NeXT ワークステーション上で行われた。Steve Jobs が Apple を追い出された後に作った会社のマシンだ。1台 $4,995〜$12,395。id Software はピザとコーラで育ったテキサスの若者たちの集団だったが、Wolfenstein 3D のシェアウェア収入で NeXT を買えるほどの稼ぎを手にしていた。

エンジンは C で書かれ、レベルエディタは Objective-C で書かれた。Carmack は NeXTSTEP について「eye-opener だった」と語っている。WorldWideWeb（鑑定書 #039）も NeXT 上で生まれた。**Jobs が失敗したと思われていた NeXT は、実は Web と DOOM の両方を生んでいた。**

1993年12月10日、DOOM はシェアウェアとしてリリースされた。最初の24時間でインターネット回線がパンクした。1994年半ばまでに、Carmack はフェラーリを2台買った。328 と テスタロッサ。

## 発掘された痕跡

### 痕跡1：BSP ツリー——学術論文がゲームエンジンになった日

```c
// DESCRIPTION:
//	BSP traversal, handling of LineSegs for rendering.
```

`r_bsp.c`——DOOM のレンダリングの心臓部。**BSP（Binary Space Partitioning）ツリー** によるシーン分割。

初期の DOOM レンダラーは、プレイヤーの現在の部屋から隣接する部屋へ「フラッディング」しながら描画していた。複雑なレベルでは同じセクターを何度も訪問し、パフォーマンスが崩壊した。

Carmack は SNES 版 Wolfenstein 3D を移植する際に BSP ツリーを研究した。1980年の Fuchs-Kedem-Naylor の論文、1991年の Chen-Gordon の論文、1993年の Naylor の論文。**学術論文を読んでゲームエンジンに実装する——Carmack の真の天才はここにある。**

学術論文は「奥から手前へ」描く方法を提案していた。Carmack は逆転させた。**「手前から奥へ」** 描く。既に描画された領域をスキップすることで、VGA の描画コストを削減する。暗黙的な Z バッファ。

レベルエディタが BSP ツリーを事前計算する。1レベルあたり約8秒。この前処理が、リアルタイムの高速描画を可能にした。

### 痕跡2：「MONDO HACK!」と「HACK ALERT!」

```c
// MONDO HACK!
```

```c
// HACK ALERT!
```

`wi_stuff.c`（インターミッション画面）に、開発者の叫びが残されている。エピソード1のアニメーション処理で、本来は専用のアセットを作るべきところを、**別のエピソードのアニメーションフレームを使い回している。**

さらに：

```c
// Time sucks.
```

時間表示の処理に対する一言。そして：

```c
// Yuck.
```

フランス語ローカライゼーション処理への感想。**開発者の正直な感情が、コメントに刻まれている。**

### 痕跡3：「Bug, dear Shawn.」

```c
// Bug, dear Shawn.
// Insufficient malloc, caused spurious realloc errors.
```

`d_main.c` に残された、**同僚 Shawn Green への手紙のようなバグ報告。** メモリ確保が不足していて、偽のメモリ再確保エラーを引き起こしていた。

Shawn Green は id Software のレベルデザイナー。Carmack がコードの中で同僚に語りかけている。**ソースコードは、チームのコミュニケーション手段でもあった。**

### 痕跡4：「This is a kludgy mess.」

```c
// This is a kludgy mess.
void P_SlideMove (mobj_t* mo)
```

`p_map.c`——壁に沿ってスライド移動する処理。**Carmack 自身が「ごちゃごちゃだ」と認めている。** Wolfenstein 3D（鑑定書 #033）が90度の壁だけだったのに対し、DOOM は任意の角度の壁がある。壁沿いのスライド計算は幾何学的に複雑になり、エレガントな解法を見つけられなかった。

しかしこの「kludgy mess」は **動いた。** 何百万人のプレイヤーが、この「ごちゃごちゃな」コードの上で壁に沿って走り続けた。

### 痕跡5：ゾーンメモリ——DOOM が自前でメモリ管理した理由

```c
// DESCRIPTION:
//	Zone Memory Allocation. Neat.
```

`z_zone.c`——DOOM 独自のメモリアロケータ。DOS のメモリ管理関数を信頼せず、**起動時に `malloc` で巨大なブロックを一括確保し、以降は自前で分割・管理する。**

ゾーン ID `0x1d4a11` がヘッダーに書き込まれ、メモリ破壊を検出する歩哨として機能する。バッファオーバーフローが起きると「Freed a pointer without ZONEID」というメッセージが出る。**1993年に、独自のメモリデバッグ機構を持っていた。**

Carmack 自身、このゾーンメモリが「Quake に持ち越せた唯一のもの」と評している。Photoshop 1.0（鑑定書 #042）が仮想メモリを自前実装したように、**OS を信頼できない時代のプログラマは、全てを自分で作った。**

### 痕跡6：Carmack の自己批評——README に残された反省

Carmack は1997年のソースコード公開時に、README で自作コードを批評している：

レンダリングの基本概念——水平・垂直方向の定Z ラインと固定光シェーディング——は「dead-on（完璧）」だったが、実装には改善の余地があった。

**最大の反省**: 移動と視線チェックに BSP ツリーを使わなかったこと。レンダリングには BSP を使いながら、衝突判定には別のシステムを使っていた。Carmack は「one of the bigger misses（大きな見落としの一つ）」と呼んでいる。

そして楽観的な予測：「最新のハードウェア（高速 Pentium + 3DFX）なら、賢くなくてもいい——レベル全体を描画しても60fps で動くだろう」。**1997年に書かれた予測が、2026年の現実になった。**

## 二人の John

DOOM は **二人の John** が作った。

**John Carmack**——プログラマー。内向的、寡黙、数学と論文を愛する。BSP ツリーを学術論文から実装に落とし込む天才。Wolfenstein 3D の後、チームから隔離して新エンジンの開発に没頭した。斜めの壁、可変照明、さらには傾斜床まで実装した（後に削除）。

**John Romero**——デザイナー。カリスマ的、派手、レベルデザインの天才。しかし彼の複雑なレベル設計が初期エンジンの性能問題を引き起こし、Carmack は BSP ツリーの研究を始めた。

二人は後に対立し、Romero は id Software を去る。しかし DOOM は、**技術の Carmack とデザインの Romero が完璧に噛み合った瞬間** だった。

## Wolfenstein 3D から DOOM へ

Wolfenstein 3D（鑑定書 #033）は「扉」だった。DOOM は、その扉の先だ。

| | Wolfenstein 3D | DOOM |
|---|---|---|
| 壁の角度 | 90度のみ | 任意の角度 |
| 床・天井 | 単色 | テクスチャマッピング |
| 照明 | 均一 | 距離減衰・セクター別 |
| 高低差 | なし | あり |
| レンダリング | レイキャスティング | BSP ツリー |
| マップ構造 | グリッドベース | 任意のポリゴン |
| 開発環境 | DOS PC | NeXT ワークステーション |

**技術的には全く異なるエンジン。** しかし Carmack が BSP ツリーを最初に試したのは SNES 版 Wolfenstein 3D だった。前作の移植が、次作の革命を生んだ。

## 推定される経緯

**1992年11月**: DOOM 開発開始。Carmack、新エンジンの設計に着手。

**1993年前半**: Carmack が BSP ツリーを SNES 版 Wolfenstein 3D で実験。DOOM レンダラーに応用。

**1993年5月**: v0.5 アルファ版。まだ BSP ツリー未実装。

**1993年10月**: プレスリリース用ベータ版。BSP ツリー実装済み。

**1993年12月10日**: DOOM 1.0 シェアウェアリリース。インターネット回線がパンク。

**1994年**: id Software の年収は数百万ドルに。Carmack がフェラーリ 328 とテスタロッサを購入。

**1997年12月23日**: ソースコード公開（GPL v2）。Carmack の自己批評付き README 同梱。

**2026年現在**: DOOM は妊娠検査薬、ATM、トースター、IKEA のスマートランプ上で動作確認されている。「Can it run DOOM?」はインターネットミームになった。

### 希少度: ★★★☆☆
C言語。GPL v2 で公開済み。GitHub に18,400スター。コード自体は広く流通しているが、1993年の「386 で動く BSP ツリー実装」としては唯一無二。

## 鑑定人所見

DOOM は「炎」だ。

Wolfenstein 3D（鑑定書 #033）が「扉」なら、DOOM はその扉の向こうで燃え盛る **炎** だ。BSP ツリーという学術的な燃料に、Carmack の実装力という火花が散り、FPS というジャンルそのものが燃え広がった。

最も象徴的なのは **「MONDO HACK!」** だ。Carmack のコードには、完璧主義者と実用主義者の二面性がある。BSP ツリーの実装は学術論文を超える精度で行われている。しかしインターミッション画面のアニメーションは「MONDO HACK!」と叫びながら別エピソードのアセットを使い回す。壁スライドの計算は「This is a kludgy mess.」と自嘲する。

**完璧にすべきところは完璧にし、そうでないところは「動けばいい」で通す。** この判断が、13ヶ月でゲーム史を変えるエンジンを完成させた秘訣だ。

PDP-7 Unix（鑑定書 #043）は「種子」だった——3,000行から全ての OS が生まれた。DOOM は **「炎」** だ——39,000行の C コードが、FPS というジャンルを燃え上がらせ、「Can it run DOOM?」は2026年の今もインターネットで燃え続けている。
