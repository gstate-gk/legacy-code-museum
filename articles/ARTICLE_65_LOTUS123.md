# VisiCalcを1年で葬った——Lotus 1-2-3が証明したキラーアプリの法則

## はじめに

1983年1月26日、Lotus 1-2-3が出荷された。

**価格 $495。** VisiCalcの約2倍。それでも飛ぶように売れた。

その年の12月、PC World誌が衝撃的な数字を報じた。VisiCalcの月間販売本数が39,000本から**5,700本**に崩落していた。11ヶ月で市場の85%が消えた。

Jon Sachs（ジョナサン・サックス）とMitch Kapor（ミッチ・カプラー）は、計算していた。IBM PCはApple IIではない。Intel 8086の640KBメモリを使い切れば、VisiCalcが逆立ちしてもできないことができる。

「**スプレッドシート、グラフ、データベースを一本のプログラムで。**」——そのコンセプトを85KBに詰め込んだのが Lotus 1-2-3 だった。

名前の由来は、Kapor の瞑想習慣。「蓮華座（lotus position）」から。プログラマーの相棒にそんな名前をつけた男が、世界で最初の「キラーアプリ」を生み出した。

## 発掘された痕跡

オリジナルのソースコードは IBM の金庫の中に眠っている。1995年の買収以来、一度も公開されていない。しかし2015年、セキュリティ研究者 Tavis Ormandy は Lotus 1-2-3 UNIX版のバイナリを解剖した。COFF形式のオブジェクトファイルに、開発者たちが消し忘れた**約20,000個のシンボル名**が残っていた。

### 痕跡1：`init_at_funcs()` ——@の王国

```c
/* 123elf / main.c より（バイナリから復元されたシンボル） */
init_at_funcs();      /* @関数ライブラリを初期化 */
```

Lotus 1-2-3 が世界に残した最大の遺産の一つは、**@ 記号** だ。

```
@SUM(A1..A10)      → 範囲合計
@AVG(B1..B12)      → 平均
@IF(C1>100,D1,E1)  → 条件分岐
@NPV(0.1,A1..A12)  → 正味現在価値
@VLOOKUP(x,range,col) → 縦方向検索
```

当時のスプレッドシートに組み込み関数の標準文法はなかった。Sachs は @ 記号を「関数の印」として採用した——アルファベットでも数字でも記号でもない、**一目でわかる関数マーカー**として。

Excel が Lotus 1-2-3 を駆逐した後も、@SUM の構文は ExcelのSUM()に変わり、Googleスプレッドシートに変わり、世界中の表計算ソフトに刻み込まれた。

Sachs が `init_at_funcs()` に込めた設計は、競合他社にコピーされることで**永遠になった。**

### 痕跡2：メニュー階層——裁判の証拠物件

```
/                     （スラッシュでメニュー起動）
├─ W  Worksheet       （ワークシート操作）
│  ├─ G  Global       （全体設定）
│  │  ├─ F  Format    （書式設定）
│  │  │  ├─ G  General
│  │  │  ├─ F  Fixed
│  │  │  ├─ C  Currency
│  │  │  └─ P  Percent
│  │  └─ C  Column-Width
│  └─ I  Insert
├─ R  Range           （範囲操作）
├─ C  Copy            （コピー）
├─ M  Move            （移動）
├─ F  File            （ファイル）
├─ P  Print           （印刷）
├─ G  Graph           （グラフ）
└─ Q  Quit            （終了）
```

1983年、このメニュー構造はユーザーにとって「第二の母国語」になった。毎朝 `/W G F C` と叩いて列幅を変え、`/R F C` と叩いて通貨書式を設定する。指が覚えた。

10年後、Borland International が Quattro Pro をリリースした。Lotus のメニュー構造を**そのままコピーして**。

Lotus は訴えた。「これは著作権侵害だ。」

1996年、米国最高裁は5対4で判決を下した。

**「メニューコマンドの階層は著作権の保護対象ではない。それは表現ではなく、操作方法だ。」**

Lotus の敗訴。しかしこの判決は、ソフトウェアのUIが「特許」ではなく「アイデア」として扱われる先例になった。Apple vs Samsung の遠い祖先。

### 痕跡3：自然順序再計算——VisiCalcに足りなかったもの

VisiCalc は列ごとに再計算していた。左から右、上から下。

問題があった。`B2` が `C3` を参照していて、`C3` がまだ計算されていないとき——計算順序が合わなければ、一回の再計算では正しい値が出ない。ユーザーは F9（再計算キー）を2回叩く必要があった。

Sachs は別の方法を設計した。

```
/* 自然順序再計算（Natural Order Recalculation）の概念 */

セルの依存グラフを構築:
  A1 → B1 → C1     （C1 は B1 に依存、B1 は A1 に依存）
  A2 → B1           （B1 は A2 にも依存）

変更セルから逆算:
  A1 が変わった
    → B1 を再計算  （A1, A2 が確定しているから可能）
      → C1 を再計算 （B1 が確定しているから可能）

一回のF9で全セルが正しく更新される。
```

ユーザーには見えない。ただ「速い」し「正確だ」とわかる。

VisiCalcの計算エンジンが2回必要だったところを、1-2-3は1回で片付けた。**PC World 誌は「信じられないほど速い」と評した。**

### 痕跡4：COFF シンボルが語る内部構造

Tavis Ormandy が Lotus 1-2-3 UNIX版を解析したとき、バイナリの中に開発者が消し忘れたシンボルテーブルを発見した。

```
/* 復元されたシンボル一覧（抜粋） */

グラフエンジン:
  x_disp_graph
  V3_disp_grph_compute_view
  V3_disp_grph_process

ファイルI/O:
  read_setup_file
  read_startup_ri_file
  file_path_expand
  slash_convert         ← バックスラッシュ変換（DOS互換）

キー入力:
  keyreader
  ready_to_read

日付処理:
  encode_date
  check_three_numbers

表示エンジン:
  clear_screen_buffer
  reset_screen_dimensions
```

`slash_convert` というシンボルが目を引く。DOS の `\` をUNIXの `/` に変換する関数だ。1983年のDOS版で設計されたパス処理が、1989年のUNIX版にそのまま残っていた。

20,000個のシンボルの中に、6年間の開発の痕跡が刻まれていた。

### 痕跡5：.WK1 フォーマット——敗者が残した遺産

Lotus 1-2-3 が定義した `.WKS`（バージョン1）と `.WK1`（バージョン2）のファイル形式は、業界標準になった。

```
.WK1 ファイル構造（バイナリ）:

Offset  Type   内容
------  ----   ----
0x00    Word   レコードタイプ（0x0000 = BOF）
0x02    Word   レコード長
0x04    Word   バージョン番号（0x0404 = Lotus 1-2-3 v2）
...
各セル:
0x00    Word   レコードタイプ（0x000E = 数値セル）
0x02    Word   レコード長
0x04    Byte   書式属性
0x05    Word   行番号
0x07    Word   列番号
0x09    Double 8バイト倍精度浮動小数点
```

Excel 2.0（1987年）がリリースされたとき、.WK1 形式のインポート機能を真っ先に実装した。

Lotus を倒すためには、まず Lotus のデータを読めることが必要だった。**敗者のファイル形式が、勝者の礎になった。**

### 痕跡6：`wrappers.c` が語る時代の壁

Tavis Ormandy が 123elf（Lotus 1-2-3 を現代 Linux で動かすプロジェクト）を作るにあたって、最も苦労したのはシステムコールの変換だった。

```c
/* 123elf/wrappers.c より */

/* I think these are all the flags 123 uses */
static const struct {
    int unix_flag;
    int linux_flag;
} open_flags[] = {
    { 1, O_WRONLY },
    { 2, O_RDWR   },
};
```

コメントが全てを語る：**「これが123が使う全フラグだと思う」。**

1989年の UNIX版 Lotus 1-2-3 は、シングルスレッド前提、静的バッファ、最小限のシステムコールで動いていた。6万行のCコードが、現代のLinux上で **35年後も動いた。**

Ormandy のコメントはこうだった：「正しい方法を知らないが、動作する実装を優先する。」

それは、Sachs と Kapor が1982年に下した判断の正確な反転だった。

## 年表

**1982年**: Mitch Kapor と Jonathan Sachs が Lotus Development Corporation を設立。資本金は Kapor がVisiCorp コンサルタント時代に得た報酬。

**1983年1月26日**: Lotus 1-2-3 Release 1 リリース。価格 $495。広告費100万ドルを投入（創業直後のスタートアップとして破格）。

**1983年10月**: PC World 報告。1-2-3の売上がVisiCalcを逆転。

**1983年12月**: VisiCalcの月間販売が5,700本に崩落（年初比85%減）。

**1984年**: Lotus Development の年間売上が $1.5億を突破。世界最大のPC用ソフトウェア企業に。

**1985年**: Jonathan Sachs が退職。Photo Window の開発へ。

**1987年**: Microsoft Excel 2.0 for Windows リリース。.WK1 インポートを実装し、Lotus ユーザーの移行を促進。

**1990年代初頭**: Windows の普及とともに Lotus 1-2-3 の市場シェアが低下。

**1994年**: Lotus Corp. v. Borland International 訴訟。Quattro Pro のメニュー階層コピーをめぐる著作権訴訟。

**1995年**: IBM が Lotus Development Corporation を**$3.5億**で買収（敵対的買収）。IBM Notes（旧Lotus Notes）の戦略的価値のため。

**1996年**: 米国最高裁、Borland 訴訟でLotus敗訴。「メニュー階層は著作権対象外」。

**2013年**: IBM、Lotus 1-2-3 の新規販売終了。

**2014年**: サポート終了。30年の幕を閉じる。

### 希少度: ★★★★★

オリジナルソースコードは IBM の管理下にあり非公開。UNIX版バイナリに残存した約20,000個のCOFFシンボルが、内部構造を垣間見せる唯一の窓口。「キラーアプリ」という概念を定義した歴史的コードとして、最高の希少度。

### 技術的負債密度: ★★★☆☆

自然順序再計算、@関数インターフェース、メニュー階層——設計はいずれも精緻だった。しかし DOS 専用最適化が後の Windows 対応を遅らせ、その隙を Excel に突かれた。**設計の優秀さが、逆に「乗り換え」の足かせになった。**

### 考古学的価値: ★★★★★

「キラーアプリ」という概念の原型。IBM PC の普及を加速させ、パーソナルコンピュータを「家庭用ゲーム機」から「ビジネスツール」に変えた。@関数の命名規則は40年後の今も生きている。最高裁の判決は「UIのアイデアは著作権対象外」という原則を確立した。

### 読み物としての面白さ: ★★★★★

VisiCalcを1年で葬った男たちが、10年後に Excel に同じことをされた。$35億で買収された企業が最終的に「サポート終了」で消えていった。Sachs が設計したメニュー階層が著作権裁判の証拠物件になり、最高裁で「著作権対象外」と宣告された。皮肉が積み重なる。

## 鑑定人所見

Lotus 1-2-3 は「嵐」だ。

1983年の冬、嵐のように来て、一年でVisiCalcを葬った。85KBに三つの機能を詰め込んで、「こんなに速いはずがない」とユーザーを驚かせた。Jonathan Sachs が設計した自然順序再計算は、誰も気づかないところで正確に動き続けた。

しかしその成功が、次の嵐への警戒を鈍らせた。

DOS 専用の最適化は Windows の時代に重石になった。メニュー階層は「業界標準」になりすぎて、新しいUIを試せなくなった。Excel はそこを突いた——Lotus の .WK1 形式を読み込み、Lotus のユーザーを引き抜き、Lotus のいなくなった場所に居座った。

Kapor が会社名を「Lotus」にしたのは、蓮の花は泥の中から美しく咲くからだと言われる。しかし蓮の花は、嵐には弱い。

**VisiCalcを葬った嵐は、より大きな嵐に葬られた。**

それが Lotus 1-2-3 が証明したキラーアプリの法則だった。
